# 项目接口、类、函数关系梳理（学习版）

> 适用项目：`word learning system 2`
>
> 目标：帮助你从“接口概念”出发，看懂当前项目的架构与设计演进。

---

## 1. 一句话总览

这个项目不是一开始就完整设计成现在这样，而是经历了：

- 基础版（只做词汇查询和简单练习）
- 扩展版（加入学习过程记录）
- 第二阶段（加入认证、数据隔离、自适应推荐、复习调度）

当前主干建议以 `app_phase2.py` + 前端 `src/lib/apiClient.ts` / `src/lib/types.ts` 为准。

---

## 2. 关键文件定位

### 后端核心

- 主入口与路由：`app_phase2.py`
- 推荐与间隔重复算法：`adaptive_engine.py`
- 认证模块：`auth.py`
- 数据隔离与权限控制：`user_data_isolation.py`
- 易混淆词接口扩展：`confusable_api.py`

### 前端核心

- API 请求封装层：`src/lib/apiClient.ts`
- TS 类型接口定义：`src/lib/types.ts`
- 用户上下文：`src/context/LearningContext.tsx`
- 练习页（完整调用链样例）：`pages/exercise.tsx`
- 时间追踪 Hook：`hooks/useTimeTracking.ts`

## 3. “接口”在这个项目里的三层含义

## 3.1 HTTP 接口（后端 API）

例如：

- `GET /words`
- `GET /word/<id>`
- `POST /api/learning/session/start`
- `POST /api/learning/exercise/record`
- `GET /api/adaptive/recommendation/<user_id>`
- `POST /api/auth/login`

这些是前后端通信的运行时接口。

## 3.2 TypeScript 类型接口（前端契约）

例如：

- `WordResponse`
- `ExerciseSetResponse`
- `UserProfileSummary`

定义在 `src/lib/types.ts`，用于约束后端返回结构。

## 3.3 Hook/函数接口（前端模块间契约）

例如：

- `UseTimeTrackingReturn`
- `UseExerciseTimerReturn`

定义在 `hooks/useTimeTracking.ts`，用于组件与逻辑模块之间的调用边界。

## 5. 数据模型类关系（后端）

```text
Word (词)
 ├─ Example[*]        (例句)
 ├─ Collocation[*]    (搭配)
 └─ Character[*]      (字)

LearningSession (某用户某词的一次学习会话)
 ├─ user_id
 ├─ word_id -> Word.id
 └─ session_id (唯一)

ExerciseRecord (每道题作答记录)
 └─ session_id -> LearningSession.session_id

UserProgress (用户-词汇长期状态)
 ├─ user_id
 ├─ word_id -> Word.id
 ├─ mastery_level / next_review_suggested
 └─ 统计字段(correct_attempts, total_attempts...)

AdaptiveRecommendation (推荐记录)
 ├─ user_id
 ├─ target_word_id -> Word.id
 └─ recommendation_data / confidence_score
```

### 说明

- `LearningSession` 偏“过程数据”（一次学习过程）
- `ExerciseRecord` 偏“明细数据”（每道题）
- `UserProgress` 偏“聚合状态”（长期掌握情况）

## 6. API → 函数 → 类 映射（核心）

## 6.1 学习会话

- `POST /api/learning/session/start`
  - 函数：`start_learning_session`
  - 行为：创建 `LearningSession`

- `POST /api/learning/session/end`
  - 函数：`end_learning_session`
  - 行为：更新结束时间、时长、完成状态

## 6.2 练习

- `GET /api/learning/word/<word_id>/exercises`
  - 函数：`get_word_exercises`
  - 行为：读取 `Word`/`Example`/`Collocation` 组题

- `POST /api/learning/exercise/record`
  - 函数：`record_exercise_result`
  - 行为：写 `ExerciseRecord`
  - 后续：调用 `SpacedRepetitionAlgorithm.update_user_progress_after_exercise` 更新复习计划

## 6.3 推荐与复习

- `GET /api/adaptive/recommendation/<user_id>`
  - 函数：`get_adaptive_recommendation`
  - 调用：`AdaptiveRecommendationEngine.get_next_recommendation`
  - 存档：`save_recommendation`

- `GET /api/review/user/<user_id>/due`
  - 函数：`get_due_reviews`
  - 调用：`SpacedRepetitionAlgorithm.get_due_reviews`

## 6.4 认证与权限

- `POST /api/auth/login`
  - 函数：`login`
  - 调用：`AuthManager.login`

- 受保护接口上的装饰器：
  - `require_authentication(...)`
  - `check_data_ownership(...)`

行为核心：用户只能访问自己的数据。

## 7. 三条最重要调用链

## 7.1 做题链路

1. 前端练习页加载题目（`pages/exercise.tsx`）
2. 调用 `fetchWordExercises`（`src/lib/apiClient.ts`）
3. 后端 `get_word_exercises` 返回题目集（`app_phase2.py`）
4. 用户提交后调用追踪/记录
5. 后端 `record_exercise_result` 入库
6. 触发 SRS 更新 `UserProgress.next_review_suggested`

## 7.2 推荐链路

1. 前端 dashboard 请求推荐
2. `buildApiUrl` 拼接地址
3. 后端 `get_adaptive_recommendation`
4. 推荐引擎 `get_next_recommendation`
5. 推荐结果落库 `adaptive_recommendation`

## 7.3 认证隔离链路

1. 前端登录拿到 `session_token`
2. 请求时通过 header/cookie 携带 token
3. 装饰器解析当前用户
4. 路由函数中对 `user_id` 做所有权校验
5. 不匹配返回 `403 PERMISSION_DENIED`

## 8. 为什么不是“一开始就这么设计”？

这是典型的工程演进：

1. **先跑通核心业务**（查词、学习、做题）
2. **再做可观测**（会话记录、题目记录、统计）
3. **再做个性化**（推荐、间隔重复）
4. **再做多用户安全**（认证、隔离）

所以现在看到的是“需求驱动演进后的架构”，而不是“一次性完美设计”。

---

## 9. 面试/答辩可直接复述模板（2分钟）

可以这样说：

> 这个项目采用前后端分层设计。前端通过 `apiClient` 调用 Flask API，并由 TypeScript 接口约束返回结构。后端以 `app_phase2.py` 为路由入口，分别把认证、权限隔离、推荐算法、间隔重复算法拆到独立模块。数据层用 `LearningSession`、`ExerciseRecord` 记录过程与明细，用 `UserProgress` 做聚合状态，用 `AdaptiveRecommendation` 存推荐历史。整体是从基础功能逐步演进到支持个性化与多用户安全的架构。

## 10. 学习建议（接口概念）

你可以按这个顺序复习：

1. **先懂 HTTP 接口**（路由、请求、响应）
2. **再懂数据接口契约**（TS interface / JSON 结构）
3. **再懂函数接口**（模块输入输出）
4. **最后看类职责**（AuthManager、RecommendationEngine、SRS）

当你能把“某个页面点击后，经过哪些接口/函数/类，最终写到哪张表”说清楚，你就真正掌握接口设计了。
